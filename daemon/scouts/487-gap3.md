## The hole today

When upstream returns 202 with body like:

```json
{
  "classifiedId": "6cc36734-c270-4f9d-bcb7-756898e3a4aa",
  "paymentId": "pay_2b9f7823f90d4213ab9b815eb514a6f8",
  "paymentStatus": "pending",
  "status": "queued",
  "checkStatusUrl": "https://x402-relay.aibtc.com/payment/pay_2b9f7823f90d4213ab9b815eb514a6f8"
}
```

The tool returns this verbatim to caller (L390-394 just spreads `response.data` into the result). Caller sees `pending` + `queued`, assumes things are normal, walks away.

But the relay can be `held` (sender nonce gap, hold reason, etc.) — info that's only available by manually following `checkStatusUrl`. Caller never knows.

## Existing scaffolding (already there)

`resolveCanonicalPaymentStatus` in `src/utils/x402-payment-state.ts` L199 — fetches canonical status via checkStatusUrl, parses with `HttpPaymentStatusResponseSchema`, returns `HttpPaymentStatusResponse | null`. It's currently called only in the **402-retry-failure** path (Interceptor 2 catch block, `x402.service.ts` L568) — never on the success path.

`classifyCanonicalPaymentStatus` produces a `decision.summary` text useful for surfacing held-state to caller.

## What Gap 3 adds

1. **Detect 202-with-paymentId in success path** (~15 lines) — sniff `response.status === 202` AND `response.data.paymentId` AND `response.data.checkStatusUrl`.

2. **Call `resolveCanonicalPaymentStatus` on the success body's checkStatusUrl** (~30 lines) — wrap in a poll-N-times helper (e.g., 3 polls over 30s, exponential backoff). Stop polling early if status reaches a terminal state (`settled` | `failed` | `dropped`).

3. **Format `relayState` + `holdReason` + diagnostic-chain hint** (~30 lines) — reuse Gap 2's structured `payment` block. New fields:
   - `relayState`: `held` | `queued` | `settled` | `failed` | `pending`
   - `holdReason`: `gap` | `expired` | `validation` | `unknown`
   - `holdExpiresAt`: ISO-8601 string when relay will release or drop
   - `nextStep`: text suggesting `nonce_health` → `check_relay_health` → file relay bug

4. **Decide return shape** (~20 lines) — option A: wait for terminal state then return; option B: return early with held-state surfaced + caller can re-query. **Recommended:** option B with a 30s soft timeout. Caller gets fast response with full visibility.

5. **Tests** (~80-100 lines) — 4-6 cases:
   - 202 + paymentId + checkStatusUrl returns settled within poll window → poll surfaces settled
   - 202 + paymentId + checkStatusUrl returns held + reason gap → poll surfaces held + nextStep hint
   - 202 + paymentId + checkStatusUrl returns 5xx (relay unavailable) → fallback to original 202 body + warning
   - 202 without paymentId (free endpoint with 202) → no polling, return as-is
   - 200 success (typical case) → no polling, return as-is
   - 202 + paymentId reaches terminal state in poll #2 (out of 3) → return early with confirmed state
