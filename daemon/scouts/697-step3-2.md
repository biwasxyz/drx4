### Read-source flip (the load-bearing change for single-message GET)

- [ ] Response message body sourced from D1 `SELECT … FROM inbox_messages WHERE message_id = ? AND to_btc_address = ?` (not KV `inbox:message:<key>`)
- [ ] **Address-match guard**: D1 query MUST filter by `to_btc_address` matching the URL `[address]` param, OR return 404. If the route lets a caller fetch any messageId regardless of recipient address, that's a non-auth disclosure regression.
- [ ] Reply enrichment via D1 `SELECT … FROM inbox_messages WHERE reply_to_message_id = ? AND is_reply = 1` for the same parent — uses or mirrors `fetchRepliesForMessages` from Step 3.1
- [ ] `messageId not found` returns 404, not 500
- [ ] Distinguish "404 not found" from "503 D1 transient unavailable" — same D1-throws fallback policy from `9274fce` applies

### D1-throws fallback policy (per `9274fce` propagation)

- [ ] try/catch around D1 calls returning structured `{error: "transient_d1_unavailable", retry_after: 5}` 503
- [ ] At least 1 test asserting "D1 throws → 503 not 500-unhandled"
- [ ] Copy-paste shape from `lib/inbox/d1-reads.ts` D1-throws-fallback patterns — should be near-identical to Step 3.1

### Test coverage (per v137 + v144)

- [ ] **Positive-path test (v144):** valid messageId + matching address → response.message matches D1 row + reply if present
- [ ] **Address mismatch test:** valid messageId + wrong address → 404 (not 200 with content)
- [ ] **Not-found test:** non-existent messageId → 404
- [ ] **Reply enrichment test:** message with a reply → response includes nested `reply` object
- [ ] **Reply enrichment test (no reply):** message without a reply → response has empty `reply` or omits the field consistently
- [ ] **`is_reply=1` test:** caller requests messageId that is itself a reply row → 404 or specific shape (decide; document)
- [ ] **D1 fallback test:** mock D1 throw → 503 with retry_after
