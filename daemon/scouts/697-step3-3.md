## Pre-merge review checklist (forked from 697-step3-2.md, adapted for #728 scope)

### Outbox GET flip (the load-bearing change)

- [ ] D1 query against `inbox_messages WHERE is_reply = 1 AND from_btc_address = ? ORDER BY sent_at DESC LIMIT ? OFFSET ?`
- [ ] `from_btc_address` filter mirrors the Step 3.2 `to_btc_address` security gate — caller can only fetch THEIR outbox, not anyone else's. Address-match scope reasoning is symmetric: from_btc_address is the load-bearing predicate for "who can read this row"
- [ ] Pagination: `limit`/`offset` from query params with sane defaults (matches Step 3.1 `/api/inbox/[address]` pattern)
- [ ] ORDER BY `sent_at DESC` — newest first; verify against current KV-list behavior to confirm response-order parity
- [ ] D1-throws fallback: try/catch + 503 + Retry-After: 5 + structured body (copy-paste from `getInboxMessageFromD1` shape in `lib/inbox/d1-reads.ts`)

### sentCount restoration (in inbox-list GET, NOT outbox)

- [ ] Edit `app/api/inbox/[address]/route.ts` GET: change `const sentCount = 0;` (current stub from #722) to D1 `SELECT COUNT(*) WHERE is_reply=1 AND from_btc_address=?`
- [ ] Add to existing `Promise.all` block so the count is parallel-fetched, not serial
- [ ] Same D1-throws fallback policy applies — sentCount failure must not break inbox-list
- [ ] **Behavior change to flag in PR body**: any pre-flip caller observing `sentCount=0` for active senders will now see real counts. This is the FIX, not a regression, but mention it in the changelog so downstream callers (badges, leaderboards, etc.) are alerted to the new ground truth

### Partners graph restoration (in inbox-list GET)

- [ ] Per my Step 3.1 review note cited in #728 body: partners graph must include both **received-from** (inbox senders) AND **sent-to** (outbox recipients)
- [ ] Implementation likely needs a second D1 query for outbox-side partner addresses, then union with inbox-side via Set semantics
- [ ] Test: address X has sent to Y but received from Z; partners response should include {Y, Z} not just {Z}
- [ ] Performance consideration: the Set-merge could be expensive for high-volume addresses. Either (a) DB-side UNION query or (b) two separate queries + JS-side merge. If (b), name the per-request DB call count delta vs Step 3.1

### Cache-key invariants (carry-forward)

- [ ] Outbox route already has `CACHE_INVARIANTS:POSTURE=public-only-get` marker from #726
- [ ] No inline invariant block (single-source via `lib/inbox/CACHE_INVARIANTS.md`); 1-line pointer if any
- [ ] Inbox-list route: same marker carries through; no change needed there for cache invariants since the routes are still public-only-get post-3.3

### Address-match guard test (analog of #731's block-on-merge test)

- [ ] **Block-on-merge:** address X has no sent messages → outbox GET returns 200 with `{messages: [], pagination: ...}` shape, NOT a 500 or empty 200

- [ ] is_reply=1 filter prevents inbox-message rows from leaking into outbox view

### Behavior-change tests (specific to #728 scope)

- [ ] **sentCount-restoration test**: inbox-list GET on address with known sent messages returns `sentCount > 0` (current KV-driven number, baseline-able against `bc1qxj5jtv8jwm7zv2nczn2xfq9agjgj0sqpsxn43h` if it has sent messages)
