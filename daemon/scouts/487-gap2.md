## Existing scaffolding (already there)

L411-451 — **canonical payment status path:** when `axiosError.x402PaymentStatus` is set, surface canonical status via `formatCanonicalPaymentStatus`. Pulls fallback paymentId + fallback txid via signature extraction. Polls transaction confirmation. Adds `Operational fallback only:` block when fallback ≠ canonical.

L453-481 — **signature-only fallback path:** when canonical missing but payment-signature header exists, extract txid, poll, surface "txid recovery fallback" message.

L484 — **generic propagation:** `return formatEndpointError(error, label);`

The error response shape today is a single `text` content block. The fallback surfaces are appended as plain text.

## What Gap 2 adds beyond the existing scaffolding

1. **HTTP status classifier** — new helper that maps response status + body to a category: `validation_error` | `auth_failure` | `missing_field` | `not_found` | `server_error` | `unknown`. The "Could not determine address from payment" error from the #487 incident is a `validation_error` — caller could retry with `btc_address` field.

2. **Structured `payment` block** in tool response — instead of (or in addition to) the appended text fallback, return a structured JSON object:
   ```
   {
     payment: {
       txid, amount_sats, asset, recipient, settled, settledAt,
       explorer, recoveryHint
     }
   }
   ```
   This is the meta-fix from the issue body — single block lets every caller verify+recover regardless of upstream API quirks.

3. **Recovery action generator** — based on classifier output, produce a list of concrete next steps:
   - validation_error → "retry with corrected body (warn: charges again)" + show the missing/invalid field
   - auth_failure → "check x402 payment authentication"
   - missing_field → "add field X to body and retry"
   - not_found → "endpoint not found; verify URL"
   - server_error → "endpoint internal error; do not retry"
   - unknown → "manual investigation needed"

4. **Don't overload `formatEndpointError`** — keep the new path as a sibling of the canonical-status / signature-fallback paths, gated on "payment-attempted AND 4xx response."
